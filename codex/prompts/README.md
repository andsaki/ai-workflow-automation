# Codex CLI カスタムコマンド 使い方ガイド

Codex CLI で使える Git ワークフロー自動化コマンドの完全ガイドです。

## 🎯 このディレクトリについて

このディレクトリは **Codex CLI 用のプロンプトファイル** を管理します。

### 重要な設計思想

- ✅ このディレクトリには **プロンプト定義ファイル** のみを管理します
- ✅ プロンプトを実行すると、**開発リポジトリ** にワークフローファイルやコードを生成します
- ❌ このディレクトリ自体に `.github/workflows/` や実装コードは置きません

### 具体例

例えば `setup-auto-close-pr` プロンプトの場合：

```
このリポジトリ (ai-workflow-automation):
  └── codex/prompts/setup-auto-close-pr.md  ← プロンプト定義のみ

開発リポジトリ (your-project):
  └── .github/workflows/auto-close-linked-pr.yml  ← プロンプト実行で生成される
```

このアプローチにより、プロンプトを一箇所で管理し、複数の開発リポジトリで再利用できます。

## コマンド一覧

### 🔧 基本 Git コマンド

| コマンド | ショート | 説明 |
|---------|---------|------|
| `codex-commit` | `cxc` | AIがコミットメッセージを生成してコミット |
| `codex-push` | `cxp` | 現在のブランチをリモートにプッシュ |
| `codex-commit-push` | `cxcp` | コミット＋プッシュを一括実行 |
| `codex-create-pr` | `cxpr` | AIがPRタイトル・本文を生成してPR作成 |
| `codex-branch` | `cxb` | 変更内容から適切なブランチ名を提案・作成 |
| `codex-resume` | `cxr` | 前回のセッションを再開 |

### 🚀 Issue ワークフロー

| コマンド | ショート | 説明 |
|---------|---------|------|
| `codex-issue <番号>` | `cxi 123` | Issueを読んで実装（各ステップで確認） |
| `implement-issue <番号>` | `ii 123` | 同上（エイリアス） |
| `codex-auto-implement-issue <番号>` | `cxai 123`, `aii 123` | Issue完全自動実装 |
| `codex-close-issue <番号>` | `cxci 123` | Issueをコメント付きでクローズ |
| `close-issue <番号>` | - | 同上（エイリアス） |

### ⚙️ GitHub Actions

| コマンド | 説明 |
|---------|------|
| `codex setup-auto-close-pr` | PRがApproveされたら連動PRを自動クローズするワークフローを設定 |

## 基本的な Git ワークフロー

### パターン1: シンプルな変更

```bash
# 1. コードを編集
vim src/index.ts

# 2. コミット（AIがメッセージ生成）
cxc

# 3. プッシュ
cxp
```

### パターン2: コミット＋プッシュを一気に

```bash
# 1. コードを編集
vim src/index.ts

# 2. コミット＋プッシュ（型チェック・Lintも実行）
cxcp
```

### パターン3: PR作成まで

```bash
# 1. ブランチ名を提案してもらう
cxb

# 2. コードを編集
vim src/index.ts

# 3. コミット＋プッシュ
cxcp

# 4. PR作成
cxpr
```

## Issue ベースのワークフロー

### パターン1: 手動確認しながら実装

各ステップでAIが確認を求めます：

```bash
# Issue #123 を実装
cxi 123

# または
implement-issue 123
ii 123
```

**実行フロー**:
1. 🔍 Issue読み取り
2. 🌿 ブランチ作成（提案→確認）
3. 📝 実装計画提示（ユーザー承認待ち）
4. ⚙️ 実装
5. 🧪 テスト・ビルド
6. 💾 コミット（メッセージ確認）
7. 📤 プッシュ
8. 🔀 PR作成（タイトル・本文確認）
9. ✅ Issueコメント

**推奨用途**: 重要な変更、大規模な実装

### パターン2: 完全自動実装

確認なしで全自動実行：

```bash
# Issue #123 を完全自動で実装
cxai 123

# または
aii 123
auto-implement-issue 123
```

**実行フロー**:
1. 🔍 Issue読み取り
2. 🌿 ブランチ自動作成
3. ⚙️ 自動実装
4. 🧪 テスト・ビルド
5. 🔧 エラー自動修正（最大3回）
6. 💾 自動コミット
7. 📤 自動プッシュ
8. 🔀 自動PR作成
9. ✅ Issueコメント

**推奨用途**: 小〜中規模の変更、単純なバグ修正

### パターン3: Issue完了

実装完了後にIssueをクローズ：

```bash
# Issue #123 をクローズ
cxci 123
close-issue 123

# 対応予定なしでクローズ
cxci 123 not_planned
```

**注意**: 通常はPRマージ時に自動クローズされるため、このコマンドは不要です。

## 使い分けガイド

### 🤔 どのコマンドを使うべき？

#### シンプルな修正（1-2ファイル）

```bash
# 手動でコミット＋プッシュ
cxcp
```

#### 小さなバグ修正（Issue対応）

```bash
# 完全自動で実装
cxai 45
```

#### 新機能追加（小〜中規模）

```bash
# 確認しながら実装
cxi 123
```

#### 大規模なリファクタリング

```bash
# 確認しながら慎重に実装
cxi 78
```

#### PR作成だけ

```bash
# 既にコミット済みの場合
cxpr
```

### 📊 ワークフロー選択チャート

```
変更の規模は？
├─ 小（1-2ファイル）
│  └─ Issue番号ある？
│     ├─ Yes → cxai <番号>
│     └─ No  → cxcp
│
├─ 中（3-10ファイル）
│  └─ Issue番号ある？
│     ├─ Yes → cxi <番号>
│     └─ No  → cxb → cxcp → cxpr
│
└─ 大（10ファイル以上）
   └─ cxi <番号>（各ステップで確認）
```

## トラブルシューティング

### コマンドが見つからない

```bash
source ~/.zshrc
```

### GitHub CLIが未認証

```bash
gh auth login
```

### コミットメッセージが気に入らない

通常の `git commit` を使用するか、生成されたメッセージを編集してください。

### 自動実装が失敗した

`cxi <番号>` で手動確認しながら実装するか、手動で修正してください。

### PR作成時にエラー

ブランチがプッシュされているか確認：

```bash
git status
git push
```

### Issue番号がわからない

```bash
gh issue list
```

## よくある質問

### Q: コミットメッセージは編集できる？

A: はい。生成されたメッセージは実行前に確認できます。気に入らない場合は手動で `git commit` してください。

### Q: PRマージしたらIssueは自動でクローズされる？

A: はい。コミットメッセージとPR本文に `Closes #123` が含まれているため、PRマージ時に自動クローズされます。

### Q: エラーが出た場合は？

A: `cxai`（完全自動）は最大3回まで自動修正を試みます。それでも失敗したら手動修正が必要です。

### Q: 複数のIssueを同時に実装できる？

A: いいえ。1つずつ実装してください。並列で作業する場合は別々のブランチで行ってください。
